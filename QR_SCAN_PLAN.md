# QR-Scan Check-In / Check-Out

## Context

**The need.** A non-technical colleague needs to move medical devices in/out of custody quickly. The most common action by far is **one device at a time**: pick up a device, scan its QR, confirm. Bulk movement (several devices in one pass) happens occasionally and is a **nice-to-have**, not the daily path.

**Design principle — two surfaces, each does one thing:**

1. **Scan = act on *this* device.** A scan reads the device's **live state** and the user's role, then opens the *correct existing dialog* (`dlgCheckOut` or `dlgCheckIn`) pre-targeted to that device — or shows a toast if the user may not act on it. The scan is simply a faster way to reach the same single-device dialog you get by clicking a row today.
2. **Inventory list = act on *a set*.** The inventory view already has search, filtering, and sorting. Add row checkboxes + an action bar there for the occasional batch job. The user filters (e.g. "department X, Kivehető"), ticks, and confirms once. No scope inference, no session state — the existing filters already scoped the set.

This replaces the earlier "scan-driven session state machine" idea. That design paid heavy complexity (mode detection, scope lists, pick-sets, in-session mode switches, discard-and-switch modals) to optimize the bulk path — the rare one — while making the common single-device case *slower* than clicking a row. The two-surface model makes the common case the fastest and keeps bulk available where filtering already lives.

**Scan methods (both need no scanner library):**
- **Phone native camera** reads a **deep-link QR** encoding `…/#/scan/ESZ-0001`. The phone OS decodes it; tapping the popup opens the app straight into the resolved dialog. Zero scanner code.
- **Handheld USB/Bluetooth scanner gun** in keyboard-wedge mode *types* the code + Enter into a focused input. Zero scanner code.

The QR holds only a static `asset_tag`; current state is always looked up fresh on every scan.

## Hardware & Deployment Requirements

- **Phones:** any modern smartphone (iOS 11+/Android 8+) — native camera reads QR, no app install.
- **Scanner gun:** any **2D/QR-capable** USB-HID or Bluetooth-HID imager (~$25–60). 1D-only laser scanners will **not** read QR. Default keyboard-wedge mode with an Enter/CR suffix. Plugs into the storeroom PC; behaves like a keyboard.
- **QR labels:** generated by the app (offline, via `qrcode`) and printed. Each encodes the device's deep-link URL.
- **Hosting:** HTTPS URL reachable on the clinic network. Deep-link scanning needs only a reachable URL, not a secure context; HTTPS is still wanted for trust and future in-app camera use.
- **⚠ Shared-state prerequisite (flag):** state currently lives in **`localStorage`** (`eszkoznyilvantarto_state_v2`) — **per-browser**. The clinic phone and the storekeeper PC are separate data silos. **True multi-user / cross-device sync requires the real backend** (the SQL schema exists; no backend yet). On one hosted instance in one browser the feature is fully usable and demoable now. Do not promise cross-device sync until the backend lands.

## Entry points (no sidebar nav item)

The scan view is reached by the **"Beolvasás" button** on the inventory page (for scanner-gun users who need the input focused) or by **QR deep-link** (`#/scan/:tag`) from a phone. It is not a destination you navigate to and wait at, so it gets no sidebar link.

## How QR + live state works (read first)

A printed QR is a static image — it holds only a fixed string: the deep-link URL with the asset tag (`…/#/scan/ESZ-0001`). **The device's current state is never in the QR.** The app looks it up live on every scan (`getDeviceByAssetTag → deviceVM → currentState/activeReservation`), so a label never goes stale and *the scan decides what happens by reading live state* — exactly like commercial asset apps (the barcode is an ID; the DB holds the truth). In today's localStorage demo "live" = this browser; with the backend it's global.

`asset_tag` is a **required, unique** device field (`devices.asset_tag VARCHAR(64) NOT NULL`, `uq_devices_asset_tag`), captured + validated at registration. The scan resolves devices by this tag, case-insensitively (`getDeviceByAssetTag`).

## Scan resolution — the core logic (`scan.js`)

One scan → read live state → open the right single-device dialog, or refuse. This is the only "smart" part of the feature, and it's small.

```js
import * as A from '../ui/actions.js';           // dlgCheckOut, dlgCheckIn
import { getDeviceByAssetTag, currentUser, currentRole, roleAtLeast } from '../state/store.js';
import { deviceVM } from '../lib/vm.js';
import { toast } from '../ui/components.js';

function resolveScan(tag) {
  const dev = getDeviceByAssetTag(tag);
  if (!dev) return toast(`Ismeretlen azonosító: ${tag}`, 'error');

  const v = deviceVM(dev);
  const me = currentUser();
  const isStore = roleAtLeast(currentRole(), 'storekeeper');

  // 1) Blocked statuses — nobody acts via scan
  if (['Selejtezve', 'Elveszett', 'Szerviz alatt'].includes(dev.status))
    return toast(`Nem kezelhető: ${dev.status}.`, 'error');
  if (v.pending)
    return toast('Visszavétel folyamatban — raktáros megerősítésére vár.', 'error');

  // 2) Held → check-in
  if (v.holderId === me.id) return A.dlgCheckIn(dev.device_id);
  if (v.holderId !== null)
    return isStore ? A.dlgCheckIn(dev.device_id)                          // force-return
                   : toast(`Másnál van: ${v.holder?.full_name}.`, 'error');

  // 3) Reserved by someone else → only storekeeper may proceed
  const resv = v.reservation;
  if (resv && resv.reserved_by !== me.id && !isStore)
    return toast(`Lefoglalva: ${v.reservedBy?.full_name}.`, 'error');

  // 4) Free, or reserved-by-me, or storekeeper override → check-out
  //    (moveAsset's check_out branch clears the reservation automatically)
  return A.dlgCheckOut(dev.device_id);
}
```

**Resolution table:**

| Scanned device is… | `user` | `storekeeper` / `it_admin` |
|---|---|---|
| Selejtezve / Elveszett / Szerviz alatt / pending check-in | toast (status) | toast (status) |
| Held by me | `dlgCheckIn` | `dlgCheckIn` |
| Held by someone else | toast `Másnál van: <name>` | `dlgCheckIn` (force-return) |
| Reserved by someone else | toast `Lefoglalva: <name>` | `dlgCheckOut` (checkout clears the reservation) |
| Reserved by me | `dlgCheckOut` (consumes the reservation) | `dlgCheckOut` |
| Free (Kivehető, in stock) | `dlgCheckOut` | `dlgCheckOut` |

Why this is enough: the existing `dlgCheckOut`/`dlgCheckIn` already encode the role differences *inside* the dialog (a `user` checks out only to themselves; a storekeeper gets a "to whom" select), and `moveAsset` re-enforces every rule at the store level and throws `OpError` on violation. The scan only needs to pick *which* dialog to open. No new dialog code, no session.

**Optional polish (not required):** for *reserved-by-me*, open `dlgCheckOut` directly (current plan) — or offer a two-button modal **[Kivétel]** / **[Foglalás lemondása]**. For storekeeper + *reserved-by-other*, optionally confirm "lemondod a foglalást és kiveszed?" before `dlgCheckOut`. Defer unless the colleague asks.

**Input wiring** (gun + manual + deep-link all funnel through `resolveScan`):
```js
input.addEventListener('keydown', (e) => {
  if (e.key !== 'Enter') return;
  e.preventDefault();
  const raw = input.value.trim(); input.value = '';
  if (raw) resolveScan(raw);
  input.focus();                       // refocus so the next shot lands here
});
```
**Deep-link entry:** `renderScan(el, { tag })` → if `tag`, call `resolveScan(decodeURIComponent(tag))` once after wiring, so a phone-camera scan lands directly in the right dialog. After a dialog closes the user is back on the scan view with a focused input, ready for the next device — that *is* the "keep scanning" loop, without any session bookkeeping.

## Inventory bulk (`inventory.js` + `bulkActions.js` + `formBits.js`)

The occasional batch lives where filtering already does. Add to the inventory table:

- A **leading checkbox column** (`data-pick="<device_id>"`) on each row; a header "select all (filtered)" checkbox.
- A **sticky action bar**, shown only when ≥1 row is ticked. From the ticked set it computes two eligible subsets and shows a button per non-empty subset:
  - **`Kivétel (N)`** — devices that are check-out-eligible for the current role (`isFree`, and for a `user` not reserved-by-other) → `dlgBulkCheckOut(ids, { onDone })`.
  - **`Visszavétel (M)`** — devices the user may check in (`holderId === me` for a `user`; any held device for storekeeper) → `dlgBulkCheckIn(ids, { onDone })`.
- A mixed selection simply shows **both** buttons with their subset counts; each acts only on its own eligible subset. Filtering to one status first (e.g. status = Kivehető) collapses this to a single obvious button — the common batch workflow.

Each bulk dialog presents **one** dependent location→dept destination form (`destFieldsHTML` + `wireDestFields`) — a batch goes to one destination in one pass (commercial-standard; two destinations = two passes). `runBulk` loops `moveAsset` and collects per-device failures as a safety net for races; `onDone(results)` unticks succeeded rows and re-paints. Survivors keep their refreshed status.

**Check-out payload (PLURAL keys):**
```js
({ device_id, event_type: 'check_out', to_user_id: me.id,
   to_locations_id, to_departments_id,
   expected_return_date: ret ? new Date(ret) : null, notes })
```
**Check-in payload** (guard `if (!to_departments_id) { toast('Ezen a helyszínen nincs választható részleg.','error'); return false; }`; for role `user` the result is `pending`):
```js
({ device_id, event_type: 'check_in',
   to_locations_id, to_departments_id, condition_at_event, notes })
```
**Summary toast:**
```js
if (!failed.length)  toast(`${label}: ${ok.length} eszköz kész.`, 'success');
else if (!ok.length) toast(`${label} sikertelen (${failed.length}): ${failed.map(f=>f.tag).join(', ')}`, 'error');
else                 toast(`${label}: ${ok.length} kész, ${failed.length} kihagyva (${failed.map(f=>f.tag).join(', ')}).`, 'default');
```
No rollback — successes persist (append-only event log).

## QR Label Generation (`qrLabel.js`)

Encode the full deep-link so a phone camera opens the scan landing:
```js
function deepLinkFor(assetTag) {
  const base = location.origin + location.pathname.replace(/[^/]*$/, '');
  return `${base}#/scan/${encodeURIComponent(assetTag)}`;
}
```
Add **`qrcode`** (npm), **lazy-loaded** inside the dialog (`const { default: QRCode } = await import('qrcode')`), `QRCode.toDataURL(url, { width:240, margin:1 })` → `<img>`. ~20KB, bundled by Vite, offline at print time. Modal shows QR + asset_tag + model; confirm = `() => { window.print(); return false; }`. Add a `@media print` rule so only the label prints. Optional later: a `/labels` storekeeper view for full-sheet printing.

## Edge Cases & Risks

- Unknown asset_tag → error toast, input cleared + refocused.
- Held / reserved by other → role-aware toast or storekeeper override per the resolution table.
- Blocked status (Selejtezve/Elveszett/Szerviz alatt/pending) → toast, never opens a dialog.
- Scanner-gun focus stealing → refocus `#scan-input` after each Enter and on view mount.
- Bulk partial failure (race only) → `runBulk` collects per-device `OpError`; summary reports `X kész, Y kihagyva`; survivors stay ticked.
- Mixed bulk selection → both subset buttons; each acts only on its eligible subset (never mis-acts).
- iOS Safari + hash URLs → `hashchange` routing works on iOS; verify no redirect strips the hash. Test on a real iPhone.
- localStorage vs backend → per-browser silos (flagged above).

## Verification — Manual End-to-End

1. `npm run dev`; open app.
2. **Scan → checkout:** as a plain `user`, visit `#/scan/ESZ-0003` where ESZ-0003 is **free** → `dlgCheckOut` opens for that device; confirm with a destination (location→dept, raktár preselected) → device **Kiadva**, holder = me; **verify the destination dept actually applied** (plural-key fix).
3. **Scan → checkin:** scan a device **you hold** → `dlgCheckIn` opens; confirm → as `user` it goes "Visszavétel folyamatban"; verify in `/pending` as storekeeper.
4. **Gun sim / refocus:** from the inventory "Beolvasás" button, type a free device's tag + Enter → its checkout dialog opens; close it → input is refocused, type the next tag → its dialog opens. `ESZ-9999` + Enter → "Ismeretlen azonosító".
5. **Refusals:** as `user`, scan a device **held by someone else** → "Másnál van: <name>"; scan a device **reserved by someone else** → "Lefoglalva: <name>"; scan a **Szerviz alatt** device → status toast. No dialog opens in any of these.
6. **Storekeeper override:** as `storekeeper`, scan a device **held by another user** → `dlgCheckIn` opens (force-return); scan a device **reserved by another** → `dlgCheckOut` opens and checkout clears the reservation.
7. **Reserved by me:** reserve a free device, then scan it → `dlgCheckOut` opens; confirming consumes the reservation.
8. **Inventory bulk:** filter to status = Kivehető in one department → tick 3 rows → action bar shows **Kivétel (3)** → confirm one destination → "Kivétel: 3 eszköz kész." Tick a mix of free + held-by-me rows → both **Kivétel (N)** and **Visszavétel (M)** appear with correct counts.
9. **QR label:** device detail → `QR címke` → modal QR decodes to `…/#/scan/ESZ-xxxx`; `Nyomtatás` opens print dialog.
10. **iOS:** from an iPhone on the clinic HTTPS network, camera-scan a printed label → Safari opens the correct dialog for that device's live state.
